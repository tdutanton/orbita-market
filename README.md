# orbita-market

Платформа по продаже спутниковых продуктов операторам ДЗЗ

![Схема взаимодействия](/readme-resources/scheme.JPG)

Предоставляет возможности создания новых пользователей и оформления заказов через единую точку
входа (API Gateway).

## payments-service

Микросервис payments-service отвечает за пользователей и их счета - создание и хранение информации,
а также за обработку запросов на оплату.

Для упрощения тестирования сервиса в swagger в качестве user id могут вводиться любые числа или
строковые литералы.

Запускается в отдельном контейнере на порту 8081.  
Эндпоинты, через которые можно обратиться к микросервису - "/api/v1/payments/...".

Структура микросервиса:

Domain - Repository - Service - Kafka - Controller

Основной функционал микросервиса - при поступлении заявки на оплату (после создания заказа в
orders-service) обработать ее и вернуть информацию с результатами, а также обновить БД.

## orders-service

Микросервис orders-service отвечает за заказы на услуги ДЗЗ - создание заказа и получение информации
о его выполнении, хранение данных в БД.

Запускается в отдельном контейнере на порту 8082.  
Эндпоинты, через которые можно обратиться к микросервису - "/api/v1/orders/...".

Структура микросервиса:

Domain - Repository - Service - Kafka - Controller

## api-gateway

Микросервис api-gateway выполняет роль оркестратора (фасада), принимающего команды, которые могут
отправляться на оба микросервиса:

- payments-service
- orders-service
  и решает на какой из них отправить команду.

Основной функционал расположен в конфигураторе application.yml.

Микросервис запускается в отдельном контейнере на порту 8080.

## Endpoints

| `POST` | `/api/v1/payments/accounts`         | Создать счет для пользователя     |

## Соблюдение ТЗ

1. Асинхронная оплата через брокер
   Используется брокер Kafka - сообщения передаются не напрямую между сервисами, а через брокер.  
   Order Service пишет в outbox → Outbox Service отправляет в Kafka → Payment Service читает из
   Kafka.  
   Клиент не ждет, пока деньги спишутся. Он получает ответ сразу, а оплата происходит в фоне через
   Kafka.
   Order Service не зависит от доступности Payment Service (если Payment Service упал, сообщения
   копятся в Kafka).

2. Outbox/Inbox
   Outbox: атомарная запись заказа + события в одной транзакции.  
   Inbox: идемпотентная вставка через ON CONFLICT DO NOTHING.
   При создании заказа данные идут в БД orderRespository и также в outboxRepository. В kafka
   отправляются необработанные сообщения из Outbox.  
   Когда Payment Service получает сообщение из kafka, он сначала сохраняет его в inbox через
   tryInsert. Inbox защищает от двойной обработки - в tryInsert используется on conflict do
   nothing (если event_id уже есть - вставка не удастся и вернется 0 в ответ на вызов).

3. Effectively exactly-once списание по order_id  
   Идемпотентность операций позволяет достичь "фактически" ровно одноразовой отправки сообщений.
   При получении сообщения в Payment Service вызывается tryInsert - если вернется 0, то не вызовется
   processPayment - исключена повторная обработка сообщения.
   В consumer() используется @Transactional, что гарантирует откат транзакции, если processPayment
   упадет.

4. Корректный баланс при конкуренции  
   Для корректного списания денег и отображения правильного баланса используется пессимистичная
   блокировка.  
   В deposit применяется accountsRepository.findByUserIdForUpdate(userId), который использует

   ```bash 
     @Lock(LockModeType.PESSIMISTIC_WRITE)
     @Query("SELECT a FROM Account a WHERE a.userId = :userId") 
   ```

   PESSIMISTIC_WRITE - пессимистическая блокировка на запись с добавлением FOR UPDATE - при запросе
   будет захвачена блокировка на уровне БД (запрет другим транзакциям изменять или читать с
   блокировкой
   выбранные строки до завершения текущей транзакции).

   Дополнительно применена оптимистичная блокировка с применением версионирования таблиц - добавлены
   аннотации @Version в таблицах. Если FOR UPDATE не сработает, будет проверена версия таблица и
   будет
   выброшено исключение StaleObjectStateException.

## Аналитика

Для анализа данных сервисов можно подключиться к соответствующим БД и отправлять sql-запросы.  
Подключиться можно напрямую с консоли к контейнеру:

- payments-service - **docker compose exec payments-db psql -U postgres -d payments_db**
- orders-service - **docker compose exec orders-db psql -U postgres -d orders_db**

Или через любую IDE для СУБД (например, dbeaver):

- payments-service - к порту **5432**
- orders-service - к порту **5433**.

Примеры запросов лежат в **docs/analytics**.
