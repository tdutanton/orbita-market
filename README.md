## payments-service

Микросервис payments-service отвечает за пользователей и их счета - создание и хранение информации,
а также за обработку запросов на оплату.

Запускается в отдельном контейнере на порту 8081.  
Эндпоинты, через которые можно обратиться к микросервису - "/api/v1/payments/...".

Структура микросервиса:

Domain - Repository - Service - Kafka - Controller

Основной функционал микросервиса - при поступлении заявки на оплату (после создания заказа в
orders-service) обработать ее и вернуть информацию с результатами, а также обновить БД.

## orders-service

Микросервис orders-service отвечает за заказы на услуги ДЗЗ - создание заказа и получение информации
о его выполнении, хранение данных в БД.

Запускается в отдельном контейнере на порту 8082.  
Эндпоинты, через которые можно обратиться к микросервису - "/api/v1/orders/...".

Структура микросервиса:

Domain - Repository - Service - Kafka - Controller

## api-gateway

Микросервис api-gateway выполняет роль оркестратора (фасада), принимающего команды, которые могут
отправляться на оба микросервиса:

- payments-service
- orders-service
  и решает на какой из них отправить команду.

Основной функционал расположен в конфигураторе application.yml.

Микросервис запускается в отдельном контейнере на порту 8080.

**OpenAPI/Swagger** доступен после запуска:  
Payments: http://localhost:8081/swagger-ui.html  
Orders: http://localhost:8082/swagger-ui.html

## Endpoints

### Endpoints payments-service

| Метод  | Путь                                | Действие                          |
|--------|-------------------------------------|-----------------------------------|
| `POST` | `/api/v1/payments/accounts`         | Создать счет для пользователя     |
| `POST` | `/api/v1/payments/accounts/top-up`  | Пополнить счет для пользователя   |
| `GET`  | `/api/v1/payments/overview`         | Список пользователей и их балансы |
| `GET`  | `/api/v1/payments/accounts/balance` | Баланс пользователя               |

### Endpoints orders-service

| Метод  | Путь                              | Действие                           |
|--------|-----------------------------------|------------------------------------|
| `POST` | `/api/v1/orders/orders`           | Создание заказа                    |
| `GET`  | `/api/v1/orders/orders`           | Заказы по user-id                  |
| `GET`  | `/api/v1/orders/orders/{orderId}` | Заказ по id и userId (в заголовке) |

## Отказоустойчивость

CircuitBreaker и fallbackUri в **API Gateway** защищают от каскадных сбоев.

## Запуск и проверка

**Запуск**

```bash
docker-compose up -d --build
```  

**Проверка работоспособности**

```bash
http://localhost:8080/actuator/health
```  

**Посмотреть активные маршруты**

```bash
http://localhost:8080/actuator/gateway/routes
```

## Информационная безопасность

Репозиторий был просканирован при помощи **Gitleaks** и **Semgrep**.

Оба вида сканирования проведены успешно.  
По результатам анализа репозитория при помощи gitleaks установлено, что чувствительные данные в
репозиторий **не попали**.

По результатам анализа semgrep уязвимостей не выявлено.

В папке **docs/security** приведены скриншоты с результатами сканирования.
